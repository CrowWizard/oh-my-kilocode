# Tooling And Context Governance

## 目标

定义 `serena`、`memory`、`context7` 的最小使用边界，以及多 subagent 场景下的上下文治理原则。

本规则用于补足“command 负责流程、agent 负责角色、MCP 负责增强能力”这一分层下的稳定执行约束。

---

## 能力分层

- `command`：任务入口与 workflow 编排
- `agent`：角色能力
- `serena` / `memory` / `context7`：流程增强层
- `rules`：稳定治理约束

增强能力不是新的 command，也不替代 agent。

---

## Serena 使用原则

适合优先使用 `serena` 的场景：

- 代码理解
- 符号定位
- 引用分析
- 跨文件影响范围判断
- 精确替换与精确修改
- 需要避免全文 grep 猜测时

不必优先使用 `serena` 的场景：

- 已明确知道单个文件路径且只需少量阅读时
- 纯文档整理任务
- 与代码语义无关的目录说明、规则整理或文案修改

---

## Memory 使用原则

适合写入 `memory` 的内容：

- 稳定术语约定
- 项目长期边界
- 长期有效的个人偏好
- 跨多轮仍有效的结构决策
- 长任务中后续仍会反复引用的稳定状态

不适合写入 `memory` 的内容：

- 一次性任务细节
- 临时调试日志
- 短期 blocker
- 单轮执行结果
- 很快失效的局部讨论结论

使用原则：

- `memory` 不是流水日志仓库
- 只有稳定、高价值、跨轮次复用的信息才应沉淀到 `memory`

---

## Context7 使用原则

适合优先使用 `context7` 的场景：

- 框架 API 查证
- 官方推荐写法确认
- 版本差异确认
- 官方迁移说明查询

不适合优先使用 `context7` 的场景：

- 泛网络搜索
- 非官方博客汇总
- 最新新闻或公告检索
- 模糊主题研究

---

## 多 Subagent 上下文治理

### 稳定上下文

应优先放在：
- `memory`

包括：

- 术语定义
- agent / command / workflow / mode 边界
- 项目长期约定
- 个人偏好
- 长期技术选型原则

### 任务上下文

应优先放在：

- `brief-project`
- `status-overview`
- 必要时 `PLAN.md`
- 必要时 `STATUS.md`

包括：

- 当前目标
- 范围与非范围
- 当前阶段
- 最小计划
- 阻塞点
- 下一步

### 局部执行上下文

subagent 只负责：

- 局部分析
- 局部实现
- 局部审查
- 局部验证

subagent 不负责：

- 长期记忆
- 全局术语治理
- 主流程收口

---

## 最小执行要求

- 复杂任务优先先过 `brief-project`
- 长任务中途优先通过 `status-overview` 收口当前状态
- 稳定约定优先沉淀到 `memory`，不要只停留在聊天记录
- 跨文件代码理解与精确修改优先考虑 `serena`
- 框架、API、迁移说明与版本差异优先考虑 `context7`
