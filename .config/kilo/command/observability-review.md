# 工作流目标

用于在上线前、故障复盘前或平台治理期间，确认系统是否具备足够的运行可见性与告警质量。

## 模式要求

- 观测对象界定、信号覆盖审查与告警设计评估阶段可在只读模式进行
- 一旦进入日志补齐、指标埋点、trace 贯通、告警规则调整或配置写入阶段，应切换到可写执行模式
- 验证观测结果、更新运行说明与归档专项结论时，如依赖本地执行结果或文件写入，应保持可写执行模式直到 workflow 收口完成

## Workflow

### 步骤 1：界定观测对象
- 确认服务、任务、API、队列或基础设施的目标范围
- 标记关键用户路径、核心依赖、失败模式与高风险变更点

### 步骤 2：审查信号覆盖
- 检查是否同时具备指标、结构化日志与链路追踪中的必要信号
- 核对关键入口、外部调用、慢操作、异常路径是否可被观测

### 步骤 3：审查告警与 SLO
- 检查告警是否围绕可行动信号，而不是噪声指标
- 评估是否存在 SLI/SLO、错误预算、阈值依据与升级路径

### 步骤 4：审查关联与定位能力
- 确认日志、指标、trace、发布版本、实例维度之间能否关联
- 检查是否能从告警快速定位到服务、接口、依赖或时间窗口

### 步骤 5：审查运行成本与保留策略
- 评估高基数指标、日志量、采样率、保留周期与查询成本
- 识别“数据很多但不可用”或“成本过高不可持续”的设计

### 步骤 6：输出专项结论
- 输出覆盖缺口、噪声告警、定位断点与优先修复项
- 给出最小改造建议，不扩展到代码风格、架构重构或通用 review
