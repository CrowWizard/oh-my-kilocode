# 工作流目标

用于处理框架大版本升级、依赖体系替换、运行时切换或伴随数据迁移的升级工程，不适用于普通代码整理。

## 模式要求

- 升级范围盘点、兼容性分析与迁移策略制定阶段可在只读模式进行
- 一旦进入 breaking changes 修复、配置改写、迁移脚本调整、兼容层落地或验证矩阵补齐阶段，应切换到可写执行模式
- 升级验证、回退说明整理与迁移结果归档如依赖本地执行结果和文件更新，应保持可写执行模式直到 workflow 收口完成

## 工具偏好

- 官方迁移说明、版本差异与推荐写法优先使用 `context7`
- 代码路径定位、跨文件影响范围分析与精确修改优先使用 `serena`

## Workflow

### 步骤 1：盘点升级面
- 确认目标版本、当前版本、受影响依赖、运行时与外部接口
- 标记高风险项：breaking changes、废弃 API、配置变更、数据格式变化

### 步骤 2：建立迁移策略
- 选择一次性升级、分阶段升级或 expand-contract 兼容迁移方案
- 明确哪些改动需要双写、兼容层、特性开关或临时适配

### 步骤 3：处理断裂变更
- 根据官方迁移说明与项目现状，逐项修复 API、配置、构建脚本与类型定义差异
- 若涉及数据库或消息格式升级，定义前向兼容、回填与回退边界

### 步骤 4：执行验证矩阵
- 按编译、静态检查、单测、集成测试、关键路径回归的顺序验证
- 对升级后最易失效的入口补充专项验证，而不是只跑默认测试命令

### 步骤 5：建立发布与观测护栏
- 明确上线顺序、灰度策略、回滚条件与升级后必看的监控信号
- 如升级涉及数据迁移，补充迁移进度、错误率、延迟与一致性检查

### 步骤 6：输出迁移结论
- 输出升级范围、兼容性处理、剩余不兼容点、验证结果与回退方案
- 明确此 workflow 的目标是“安全升级”，不是泛化为普通 refactor
