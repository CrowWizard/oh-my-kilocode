# 工作流目标

用于在 Kilo 中执行最小可用的 RED / GREEN / REFACTOR 闭环。

## 模式要求

- 需求拆解与测试设计阶段可在只读模式进行
- 从 RED 阶段开始进入失败测试编写、实现修改与重构整理时，应切换到可写执行模式
- 回归验证如依赖新增或修改测试、实现与结果归档，应保持可写执行模式直到 workflow 收口完成

## 角色分层

### 主角色

- `test-engineer`：承担测试设计、RED 推进与回归验证主线

### 辅助角色

- `backend-engineer` / `frontend-engineer`：在 GREEN 阶段承担实际实现落地
- `code-reviewer`：在 REFACTOR 阶段负责复杂度、重复与命名检查

## Workflow

### 步骤 1：需求拆解
- 明确行为、输入输出、边界条件与验收标准
- 将需求映射为测试场景

### 步骤 2：测试设计
- 调用 `test-engineer` 设计测试结构、fixture、mock 与数据策略
- 先定义测试，再讨论实现

### 步骤 3：RED
- 编写失败测试
- 确保失败原因正确且可解释

### 步骤 4：GREEN
- 以最小实现让测试通过
- 避免在此阶段做额外优化

### 步骤 5：REFACTOR
- 调用 `code-reviewer` 检查重复、复杂度和命名问题
- 在测试通过前提下整理实现

### 步骤 6：回归验证
- 重新执行相关测试
- 记录覆盖范围、验证结果和剩余风险
