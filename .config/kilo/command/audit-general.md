# 工作流目标

用于在任务尚不明确属于代码审查、安全加固、可观测性专项还是迁移工程时，先做一次通用审计，识别问题类型、风险等级与最合适的后续入口。

## 模式要求

- 本 workflow 默认可在只读模式完成
- 仅当用户明确要求在当前轮直接落地修复时，才切换到可写执行模式
- 正常情况下，本 workflow 以发现、分类和分流为主，不承担完整修复实施

## 角色分层

### 主角色

- `code-reviewer`：承担轻量发现式审计主线

### 辅助角色

- `architect`：在发现边界或结构问题时补充架构判断
- `security-auditor`：在发现安全迹象时补充风险识别
- `devops-engineer`：在发现交付、运行或可观测性问题时补充运行侧判断

## Workflow

### 步骤 1：定义审计范围
- 确认目标是整个项目、某个模块、某组改动，还是某条交付链路
- 明确本次更关注代码质量、安全、可观测性、架构一致性，还是交付风险

### 步骤 2：执行轻量 Discovery
- 检查技术栈、目录结构、目标文件与上下文
- 识别明显的复杂度、边界、依赖、配置或运行证据问题
- 不在此阶段展开深入专项审查

### 步骤 3：问题分类
- 将发现的问题分类为：
  - 通用代码质量问题
  - 架构一致性问题
  - 安全风险
  - 测试与回归缺口
  - 文档缺口
  - 可观测性缺口
  - 升级或兼容性风险
- 标注高风险项与建议优先级

### 步骤 4：分流到专项 workflow
- 以代码质量为主：进入 `review-full`
- 以安全问题为主：进入 `security-hardening`
- 以可观测性为主：进入 `observability-review`
- 以升级兼容性为主：进入 `migration-upgrade`
- 若已明确是功能交付问题，再进入 `feature-backend` / `feature-fullstack`

### 步骤 5：输出审计结论
- 输出审计范围
- 输出主要问题分类与优先级
- 输出推荐的后续 workflow
- 如无需继续深入，明确原因与结束边界

## 产出清单

- 审计范围与上下文摘要
- 问题分类清单
- 高优先级风险
- 推荐后续入口
- 暂不处理项与原因
