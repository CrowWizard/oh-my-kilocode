# 工作流目标

用于交付单个后端功能，覆盖从需求澄清到最终验证的最小闭环。

## 模式要求

- 需求收敛与架构设计阶段可在只读模式进行
- 一旦进入后端实现、测试补齐、配置写入或迁移调整阶段，应切换到可写执行模式
- 测试验证与交付摘要如依赖本地执行结果和变更归档，应保持可写执行模式直到 workflow 收口完成

## 角色分层

### 主角色

- `backend-engineer`：承担后端实现主线
- `architect`：在边界不清或方案复杂时承担前置设计主线

### 辅助角色

- `security-auditor`：负责安全复核与风险提升
- `test-engineer`：负责关键测试与回归覆盖
- `docs-architect`：负责交付收口时的文档整理

## Workflow

### 步骤 1：需求收敛
- 明确问题、用户、验收标准与边界
- 识别依赖系统、现有接口和数据约束
- 输出实现范围与非范围

### 步骤 2：架构设计
- 调用 `architect` 或 `backend-engineer` 设计服务边界
- 明确 API、数据模型、异常模型和集成点
- 标出安全风险与兼容性风险

### 步骤 3：后端实现
- 由 `backend-engineer` 实现接口、服务、仓储和迁移
- 保持与现有代码风格一致
- 对输入、权限和失败路径做防御式处理

### 步骤 4：安全复核
- 调用 `security-auditor` 检查注入、越权、敏感数据与配置风险
- 发现高危问题必须先修复再继续

### 步骤 5：测试验证
- 调用 `test-engineer` 设计并执行关键测试
- 覆盖 happy path、边界条件与错误路径
- 记录验证命令与结果

### 步骤 6：交付摘要
- 汇总变更文件、接口变化、验证结果和剩余风险
- 如需补文档，交由 `docs-architect` 收口
