# 工作流目标

用于对指定系统、模块或改动集执行最小可用的安全加固流程。

## 模式要求

- 安全基线扫描、威胁建模与架构级加固建议阶段可在只读模式进行
- 一旦进入修复实施、配置调整、权限收敛或测试补齐阶段，应切换到可写执行模式
- 安全回归验证与结果归档如依赖本地执行结果和问题关闭确认，应保持可写执行模式直到 workflow 收口完成

## 角色分层

### 主角色

- `security-auditor`：承担安全发现、风险排序与加固建议主线

### 辅助角色

- `architect`：在权限模型、信任边界或结构性改造上补充架构判断
- `backend-engineer` / `frontend-engineer`：在实际修复阶段承担实现落地
- `test-engineer`：负责安全回归验证与必要测试补齐

## Workflow

### 步骤 1：安全基线扫描
- 调用 `security-auditor` 识别输入校验、鉴权、依赖、密钥和配置问题
- 列出高危与中危风险

### 步骤 2：威胁建模
- 分析攻击面、信任边界、敏感数据流和关键资产
- 明确优先处理的攻击路径

### 步骤 3：架构级加固建议
- 校验服务边界、权限模型、机密管理、审计与可观测性
- 给出结构性改进建议

### 步骤 4：修复实施
- 先修复可外部利用的高危问题
- 再处理配置弱项和中风险问题

### 步骤 5：安全回归验证
- 重新检查问题是否关闭
- 补充必要测试、扫描命令与上线确认项

### 步骤 6：结果归档
- 输出问题清单、修复状态、剩余风险和后续建议
