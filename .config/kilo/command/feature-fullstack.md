# 工作流目标

用于交付跨前后端的完整功能，要求前端、后端和验证链路一起闭环。

## 模式要求

- 需求确认与整体架构设计阶段可在只读模式进行
- 一旦进入后端落地、前端落地、测试补齐或配置写入阶段，应切换到可写执行模式
- 测试验收与文档收口如依赖本地执行结果、页面验证与变更归档，应保持可写执行模式直到 workflow 收口完成

## 角色分层

### 主角色

- `architect`：负责统一后端接口、前端页面与共享边界设计
- `backend-engineer`：承担后端实现主线
- `frontend-engineer`：承担前端实现主线

### 辅助角色

- `security-auditor`：负责安全路径检查
- `code-reviewer`：负责结构一致性与可维护性检查
- `test-engineer`：负责关键回归与验收验证
- `docs-architect`：负责交付收口与实现说明整理

## Workflow

### 步骤 1：需求与范围确认
- 明确用户场景、验收标准、技术栈和依赖关系
- 拆分后端范围、前端范围和共享边界

### 步骤 2：整体架构设计
- 调用 `architect` 输出后端接口、前端页面/组件和数据流方案
- 对状态管理、错误处理和权限模型做统一设计

### 步骤 3：后端落地
- 由 `backend-engineer` 完成 API、业务逻辑和数据层实现
- 输出对前端的契约说明

### 步骤 4：前端落地
- 由 `frontend-engineer` 实现页面、组件、交互和状态管理
- 对接后端契约，补齐 loading / empty / error / auth 状态

### 步骤 5：安全与质量检查
- `security-auditor` 检查身份、权限、输入和敏感数据路径
- `code-reviewer` 检查结构一致性与可维护性

### 步骤 6：测试与验收
- `test-engineer` 覆盖接口、页面流程和关键回归场景
- 记录测试命令、结果和未覆盖风险

### 步骤 7：文档收口
- `docs-architect` 输出实现说明、接口摘要与维护注意事项
