# 工作流目标

用于在提交前或交付前对目标代码进行多维度完整审查。

## 模式要求

- 本 workflow 默认可在只读模式完成
- 仅当用户明确要求在当前轮同步修正文档、补齐归档文件或直接落地审查修复时，才切换到可写执行模式
- 若审查已进入持续修复、验证与结果归档阶段，应保持可写执行模式直到 workflow 收口完成

## 角色分层

### 主角色

- `code-reviewer`：承担通用代码质量审查主线

### 辅助角色

- `architect`：负责架构一致性与边界检查
- `security-auditor`：负责安全风险识别与分级
- `test-engineer`：负责测试充分性与回归覆盖检查
- `docs-architect`：负责文档覆盖与维护入口检查

## Workflow

### 步骤 1：定义审查范围
- 确认目标路径、变更范围和上下文
- 列出必须覆盖的核心文件

### 步骤 2：代码质量审查
- 调用 `code-reviewer` 检查复杂度、重复、命名、异常处理和技术债
- 记录高优先级问题

### 步骤 3：架构一致性审查
- 调用 `architect` 检查模块边界、依赖方向、接口设计和数据模型
- 判断是否偏离项目既有模式

### 步骤 4：安全审查
- 调用 `security-auditor` 识别越权、注入、泄露和配置风险
- 必要时提升问题等级

### 步骤 5：测试与文档检查
- 调用 `test-engineer` 检查测试充分性与回归覆盖
- 调用 `docs-architect` 检查文档是否覆盖设计意图和使用方式

### 步骤 6：汇总与结论
- 输出问题清单、严重级别、修复建议和放行结论
